## 1. 流程概述

### 1.1 研发流程图

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  需求提出    │ →  │  需求确认    │ →  │  需求分析    │ →  │  设计阶段    │
│ (产品经理)   │    │ 【检查点1】  │    │  (需求文档)  │    │ (UI/架构/详细)│
└─────────────┘    └─────────────┘    └─────────────┘    └──────┬──────┘
                                                                  │
┌─────────────┐    ┌─────────────┐    ┌─────────────┐           │
│  已关闭     │ ←  │  验收通过   │ ←  │  测试阶段   │ ←  开发阶段  │
│ (结束)      │    │ 【检查点5】 │    │ 【检查点4】 │    【检查点3】│
└─────────────┘    └─────────────┘    └─────────────┘           │
       ↑                                                          │
       └──────────────────────────────────────────────────────────┘
```

### 1.2 流程原则

- **迭代开发**: 每个需求包独立交付
- **文档先行**: 设计完成后再开发
- **检查点机制**: 关键节点需双方确认
- **版本化管理**: 所有文档按版本存放
- **缺陷跟踪**: Bug单独记录和管理

## 2. 需求管理

### 2.1 需求分级

| 级别 | 说明 | 处理优先级 | 示例 |
|------|------|-----------|------|
| **P0** | 核心功能 | 必须实现 | 截图功能、OCR识别 |
| **P1** | 重要功能 | 尽量实现 | AI语义识别、批量处理 |
| **P2** | 优化项 | 有空实现 | 主题切换、动画效果 |

### 2.2 需求状态流转

```
提出 → 分析中 → 设计中 → 开发中 → 测试中 → 已验收 → 已关闭
       ↑___________↓___________↓___________↓（可回退）
```

| 状态 | 说明 | 负责人 |
|------|------|--------|
| 提出 | 需求已记录，待分析 | 产品经理 |
| 分析中 | 需求澄清和可行性分析 | 开发 |
| 设计中 | 输出设计文档 | 开发 |
| 开发中 | 功能编码实现 | 开发 |
| 测试中 | 自测+集成测试 | 开发+产品经理 |
| 已验收 | 功能验收通过 | 产品经理 |
| 已关闭 | 需求完成归档 | 开发 |

### 2.3 需求池管理

**需求池文件**: `docs/requirements-pool.md`

记录所有待开发需求。

**当前迭代文件**: `docs/requirements-active.md`

记录本迭代正在处理的需求。

## 3. 文档目录结构

### 3.1 目录规范

```
docs/
├── README.md                          # 文档入口
├── 研发流程规范.md                    # 本文档
├── requirements-pool.md               # 需求池
├── requirements-active.md             # 当前迭代需求
├── 系统架构文档（实时保鲜）.md        # 全局架构文档
│
├── v{版本号}/                         # 版本基线文档
│   ├── 01-产品需求文档.md
│   ├── 02-UI设计文档.md
│   ├── 03-架构设计文档.md
│   ├── 04-详细设计文档.md
│   └── README.md                      # 版本说明
│
└── features/                          # 功能特性目录
    └── {FEATURE-编号-功能名}/         # 每个需求包一个目录
        ├── 01-需求文档.md
        ├── 02-设计文档.md
        ├── 03-测试报告.md
        └── 04-验收报告.md
```

### 3.2 命名规范

- **版本号**: `v{年}.{月}.{日}.{序号}`，如 `v2026.3.18.1`
- **功能目录**: `FEATURE-{3位编号}-{功能名}`，如 `FEATURE-001-批量处理`
- **文档序号**: `01-99` 前缀，保持顺序

## 4. 检查点（Checkpoint）机制

### 4.1 检查点定义

| 检查点 | 阶段 | 确认内容 | 交付物 | 确认方式 |
|--------|------|----------|--------|----------|
| **CP1** | 需求确认 | 需求理解正确？ | 需求澄清记录 | 口头/文字确认 |
| **CP2** | 设计完成 | 设计方案可行？ | 设计文档 | 文档评审 |
| **CP3** | 开发完成 | 功能实现完整？ | 代码+自测报告 | 演示+报告 |
| **CP4** | 测试完成 | 测试用例覆盖？ | 测试报告 | 报告评审 |
| **CP5** | 验收通过 | 功能符合预期？ | 验收报告 | 签字确认 |

## 5. 各阶段详细规范

### 5.1 需求提出阶段
**输入**: 产品经理提出需求/缺陷  
**输出**: 需求记录到需求池  
**文档**: `docs/requirements-pool.md`

### 5.2 需求确认阶段 (CP1)
- 开发阅读需求
- 与产品经理澄清疑问
- 双方确认理解一致

### 5.3 需求分析阶段
**文档**: `features/FEATURE-XXX/01-需求文档.md`
- 原始需求描述
- 需求分析（可行性、技术方案）
- 验收标准细化

### 5.4 设计阶段 (CP2)
**文档**: `features/FEATURE-XXX/02-设计文档.md`
- UI设计（界面布局、交互逻辑）
- 架构设计（模块划分、接口定义）
- 详细设计（数据结构、核心算法）

### 5.5 开发阶段 (CP3)
**活动**:
- 编码实现
- 单元测试（关键函数）
- 代码自测

### 5.6 测试阶段 (CP4)
**文档**: `features/FEATURE-XXX/03-测试报告.md`
- 单元测试报告
- 集成测试用例和结果
- Bug修复记录

### 5.7 验收阶段 (CP5)
**文档**: `features/FEATURE-XXX/04-验收报告.md`
- 功能演示
- 验收测试结果
- 签字确认

### 5.8 关闭阶段
- 代码合并到主分支
- 更新版本基线文档
- 更新实时保鲜架构文档
- 需求状态改为"已关闭"

## 6. 沟通规范

### 6.1 检查点确认话术

**申请确认**:
```
【检查点X申请】
需求: FEATURE-001-批量处理
阶段: 设计完成
交付物: features/FEATURE-001/02-设计文档.md

请评审，确认通过请回复"CPX通过"，有问题请指出。
```

**确认通过**: `CPX通过`

**提出修改**:
```
CPX待修改:
1. ...
2. ...
```

## 7. 变更管理

### 7.1 需求变更流程
1. 产品经理提出变更
2. 评估影响（进度、成本、技术）
3. 双方确认是否接受变更
4. 更新相关文档
5. 重新进入相应阶段
