# 需求文档
## FEATURE-003: 智能打码范围优化

---

## 1. 原始需求描述

**提出日期**: 2026-03-18
**提出人**: 产品经理 (Ginsan)

**需求描述**:  
优化下打码范围，现在识别的敏感块会被全部大打码，比如"姓名：张三"，但是更好的体验是只有张三被打码，姓名不，这样不影响正常信息获取。如果类似"手机号：12311221122"可以这遮罩部分更好比如123****22。当然如果技术实现有难度就不用。

---

## 2. 需求分析

### 2.1 当前问题

**当前打码方式**:
- 整个文字块都被打码
- 例如: "姓名：张三" → 整个矩形区域被马赛克
- 问题: 标签"姓名："也被遮挡，用户无法知道这是什么信息

**期望打码方式**:
- 只打码敏感值，不打码标签
- 例如: "姓名：张三" → 只打码"张三"，保留"姓名："可见
- 效果: 用户知道这是姓名，但看不到具体值

### 2.2 技术方案分析

#### 方案1: 精确打码（推荐）
**思路**: 根据OCR结果，只打码敏感值部分

**实现方式**:
1. 在OCR阶段，记录每个文字块内敏感信息的具体位置
2. 打码时，只打码敏感值对应的子区域
3. 例如: "姓名：张三" → 只打码"张三"对应的坐标

**技术难点**:
- 需要云端视觉模型返回子坐标（目前只返回整个文字块）
- 或者后端做二次分析，识别标签和值的分界点

#### 方案2: 部分打码（手机号等）
**思路**: 对长数字串进行部分遮蔽

**示例**:
- 手机号 "13800138000" → "138****8000"
- 身份证号 "110101199001011234" → "110101********1234"
- 银行卡号 "6222 0222 0000 3333 8888" → "6222 **** **** **** 8888"

**实现方式**:
1. 根据敏感类型，决定打码比例
2. 保留前几位和后几位，中间打码
3. 可以视觉呈现为部分马赛克或黑色遮罩

**技术可行性**: 高，只需修改打码逻辑，无需改动OCR

### 2.3 推荐方案

**Phase 1（基础版本）**: 精确打码标签和值
- 修改OCR Prompt，要求返回标签和值的分界
- 或者使用规则匹配确定分界点
- 优先实现常见格式："标签：值"

**Phase 2（增强版本）**: 部分打码
- 对数字类敏感信息实现部分遮蔽
- 用户可配置遮蔽比例

**技术可行性**: 
- Phase 1 中等难度，需要调整OCR返回格式
- Phase 2 低难度，纯前端逻辑

## 3. 功能范围

### 3.1 包含的功能（Phase 1）

- ✅ **精确打码**: 只打码敏感值，保留标签可见
  - 支持格式: "标签：值"、"标签-值"、"标签 值"等
  - 常见标签: 姓名、手机号、地址、身份证号、银行卡号等
  - 示例: 
    - "姓名：张三" → 只打码"张三"
    - "手机号：13800138000" → 只打码"13800138000"
    - "地址：北京市朝阳区" → 只打码"北京市朝阳区"

- ✅ **智能分界**: 
  - 识别常见分隔符（：: - 空格等）
  - 如果无法分界，则打码整个文字块（降级方案）

### 3.2 可选功能（Phase 2，优先级低）

- ⏸️ **部分打码**: 
  - 手机号: 保留前3后2，中间打码
  - 身份证号: 保留前6后4，中间打码
  - 银行卡号: 保留前4后4，中间打码

### 3.3 不包含的功能

- ❌ 完全删除原有打码方式（作为降级方案保留）
- ❌ 所有格式都支持精确打码（仅支持常见格式）

## 6. 技术方案

### 6.1 核心改动

#### 6.1.1 数据结构修改

**当前 Region 结构**:
```python
{
    "bbox": [x1, y1, x2, y2],  # 整个文字块
    "label": "姓名",
    "text": "姓名：张三",
    "source": "ai"
}
```

**新 Region 结构（可选）**:
```python
{
    "bbox": [x1, y1, x2, y2],      # 整个文字块（用于显示标签）
    "sensitive_bbox": [x1, y1, x2, y2],  # 敏感值部分（用于打码）
    "label": "姓名",
    "text": "姓名：张三",
    "sensitive_text": "张三",       # 敏感值文本
    "source": "ai"
}
```

#### 6.1.2 精确打码算法 (core.py)

```python
def calculate_sensitive_bbox(full_bbox, full_text, sensitive_text):
    """
    根据完整文字和敏感值，计算敏感值在文字块中的相对位置
    
    方案1: 基于字符比例估算
    - 假设字符宽度均匀分布
    - 根据敏感值字符数 / 总字符数，计算比例
    - 应用到bbox上
    
    方案2: 使用OCR返回的子坐标（如果模型支持）
    """
    # 示例: "姓名：张三" (6字符), "张三" (2字符)
    # 比例: 2/6 = 0.333
    # 如果bbox宽度=120px，则敏感值宽度=40px
    # 根据敏感值位置（末尾），计算起始x坐标
    
    x1, y1, x2, y2 = full_bbox
    total_chars = len(full_text)
    sensitive_chars = len(sensitive_text)
    
    # 找到敏感值在文本中的起始位置
    start_idx = full_text.find(sensitive_text)
    if start_idx == -1:
        return full_bbox  # 找不到，返回整个bbox
    
    # 计算比例（假设字符均匀分布）
    width = x2 - x1
    char_width = width / total_chars
    
    sensitive_x1 = x1 + start_idx * char_width
    sensitive_x2 = sensitive_x1 + sensitive_chars * char_width
    
    return [int(sensitive_x1), y1, int(sensitive_x2), y2]
```

#### 6.1.3 标签-值分离逻辑 (core.py)

```python
def split_label_value(text):
    """
    分离标签和值
    
    支持的分隔符:
    - 中文冒号：
    - 英文冒号:
    - 横杠 -
    - 空格
    """
    separators = ['：', ': ', ':', '-', ' - ']
    
    for sep in separators:
        if sep in text:
            parts = text.split(sep, 1)  # 只分割一次
            if len(parts) == 2:
                label = parts[0].strip()
                value = parts[1].strip()
                # 验证标签是否在白名单中
                if is_valid_label(label):
                    return label, value
    
    return None, text  # 无法分离

# 常见标签白名单
VALID_LABELS = ['姓名', '名字', '手机号', '电话', '地址', '身份证号', '身份证', '银行卡', '邮箱', 'email']

def is_valid_label(label):
    """检查标签是否在白名单中"""
    for valid in VALID_LABELS:
        if valid in label:
            return True
    return False
```

### 6.2 实现步骤

1. **修改检测逻辑** (core.py):
   - 在 `detect_by_rules()` 和 `detect_by_ai()` 中添加标签-值分离
   - 计算 `sensitive_bbox`
   
2. **修改打码逻辑** (core.py):

## 7. 风险评估

### 7.1 技术风险

| 风险 | 可能性 | 影响 | 应对措施 |
|------|--------|------|---------|
| 字符比例估算不准确 | 中 | 打码位置偏移 | 增加padding，允许一定误差 |
| 复杂格式无法分界 | 高 | 降级为整段打码 | 接受降级方案，不破坏原有功能 |
| 性能下降 | 低 | 计算增加 | 优化算法，缓存结果 |

### 7.2 兼容性风险

- **向后兼容**: 新Region结构可选字段，不影响旧代码
- **降级方案**: 无法分界时，行为与当前一致
